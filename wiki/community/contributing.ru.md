# Руководство контрибьютора

## Кратко

Вклад в DEOS означает изменение одного домена без ослабления всего графа. Изменение может затрагивать runtime-код, экономические формулы, клиентский UX, wiki-страницы, скрипты или метаданные релиза, но дисциплина остается одной: явные контракты, детерминированное экономическое поведение, честные модели чтения и проверенные границы.

## Перед тем как писать код

### 1. Найдите домен

Сначала используйте [Карту доменов](../concepts/domain-map.ru.md). Решите, относится ли изменение к экономической физике, автономным акторам, маршрутизации, governance, стейкингу, моделям чтения, клиентскому UX, инструментам или будущей внешне-зависимой работе.

### 2. Уважайте physics-first design

DEOS предпочитает ограниченные реакции под управлением протокола ручному вмешательству. Governance может задавать направление и авторизовать изменения, но критичное для выживания экономическое поведение должно оставаться явным, детерминированным и проверенным.

Смотрите [Physics-first против politics-first](../comparisons/physics-vs-politics.ru.md).

### 3. Держите forkability в голове

DEOS — фреймворк для downstream-экосистем. Upstream-вклад должен укреплять переиспользуемые поверхности фреймворка: надежность runtime, модели чтения, проверки, инструменты, ясность docs/wiki и честность эталонного клиента. Специфическая бизнес-логика продукта обычно должна оставаться downstream.

## Рабочий процесс разработки

### Выберите правильный слой проверки

Используйте [Трехуровневую валидацию](../development/three-layer-validation.ru.md):

1. simulation для экономических формул и инвариантов;
2. проверки поведения runtime для pallet-ов и интеграций;
3. системная проверка для междоменных изменений.

Работа с клиентом также должна запускать проверочный стек web-client, а работа с wiki — проверку trusted wiki.

### Сохраняйте zero-warning hygiene

Runtime и клиентский код не должны накапливать warnings, debug leakage, unsafe type escapes или stale terminology. Если shortcut необходим, локально объясните, почему он ограничен.

### Синхронизируйте знания

Когда поведение меняется, обновляйте правильную поверхность знаний:

- Wiki-страницы для объяснений новичкам и связей между страницами;
- Backlog для открытой работы;
- Changelog для завершенных поставок;
- Durable context, если изменилось переиспользуемое правило или pattern.

## Границы хорошего вклада

Хороший вклад обычно:

- Закрывает конкретный backlog item или ясный обнаруженный срез;
- Улучшает validation или уменьшает ambiguity;
- Убирает повторяющийся UI, docs или terminology drift;
- Усиливает честность runtime/client;
- Держит работу за future gates закрытой до появления внешнего условия.

Рискованный вклад обычно:

- Добавляет feature growth без выявленного давления домена;
- Переносит unbounded history в состояние consensus;
- Делает indexers скрытой зависимостью для канонических потоков;
- Возвращает manager/farmer wording для текущих System AAA actors;
- Превращает wiki-страницы в заметки релиза или дубликат docs.
