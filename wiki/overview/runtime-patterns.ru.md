# Паттерны runtime

## Кратко

DEOS — это не только экономическая модель, но и современный runtime на базе Polkadot SDK. Проект сознательно строится на актуальных platform-паттернах и не пытается удерживать старые substrate-привычки только потому, что они когда-то были нормой.

## Подтвержденная основа runtime

Текущее устройство рабочего пространства и сборка runtime задают конкретные границы платформы:

- Рабочее пространство следует линии библиотек Polkadot SDK `2606`;
- Паллеты используют типизированную конфигурацию FRAME v2 и адаптеры, которыми владеет runtime;
- Omni Node заменяет собственный узловой crate внутри репозитория;
- XCM, идентичность активов, API runtime и веса остаются явными поверхностями интеграции;
- Планирование V3 остается отключенным до прохождения отдельной проверки готовности.

Для DEOS это не стилистика, а текущая граница платформы и набор обязательных проверок.

## Почему это важно новичку

Если читать только экономические документы, можно легко подумать, что DEOS — это просто набор особых паллетов с интересной токеномикой. Но это не так.

DEOS строится как полноценный современный runtime, который хочет оставаться совместимым с актуальной базовой линией Polkadot SDK. Это напрямую влияет на сборку, тестирование, обновления и operational safety.

## Как это связано с экономикой

Эти паттерны сами по себе инфраструктурные, но без них не будет работать и экономический слой:

- Неправильное wiring может сломать выпуск блоков
- Плохая интеграция активов ударит по маршрутизации и ликвидности
- Выбранный консенсусный и deployment-путь влияет на реальную рабочую форму launch-line
