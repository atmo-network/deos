# GCFMOS

GCFMOS is the **Genesis Creator of Feature Meta-Organ Skill**. Its presence grants standing project permission to create and evolve feature-specific delivery organs when new or substantially reshaped epics appear across validated slices, sessions, or agents.

An **FMOS** (**Feature Meta-Organ Skill**) is a bounded feature-specialist project organ for one evolving epic. It connects verified repository reality to a feature working model, canonical plan reconciliation, eligible slice selection, evidence, gates, bounded insight, handoff, and completion pull. It owns continuity of the feature-delivery feedback loop, never the implementation or truths flowing through that loop.

## Responsibility Split

| GCFMOS owns | A generated FMOS owns |
| --- | --- |
| Decide whether an epic has earned a distinct delivery organ | Maintain one feature-specific delivery feedback loop and completion pull |
| Search for a reusable existing skill before creating one | Route changed feature scope through canonical owners, commands, and checks |
| Generate the smallest valid FMOS contract | Preserve feature-specific gates, exclusions, sequencing constraints, and evidence interpretation |
| Enforce naming, ownership, dependency direction, and truth boundaries | Reconcile its method when implementation friction exposes a reusable lesson |
| Review organs for refinement, composition, split, merge, generalization, or retirement | Promote or prune bounded insight and hand off transitions without cloning the backlog |

GCFMOS never owns feature implementation, feature decisions, backlog priority, release approval, or validation semantics. It may invoke canonical validation while creating or evolving an organ. An FMOS never creates sibling organs, governs GCFMOS, rewrites unrelated plans, resolves unowned product or architecture decisions, or claims ownership of code, specification, or project-state truth.

## Protected Kernel

GCFMOS may evolve aggressively around its edges, but every mutation must preserve this kernel:

1. **Creational identity**: GCFMOS creates and gardens feature organs; it never becomes the organ for a feature.
2. **Child autonomy**: Each FMOS owns its feature feedback loop; no child governs the parent or sibling organs.
3. **Truth separation**: Host-project surfaces retain implementation, open-work, architecture, history, execution, and decision ownership.
4. **Authority preservation**: An FMOS selects only canonically eligible work; it never manufactures priority or silently arbitrates conflicting owners.
5. **Standing local permission**: Parent presence authorizes proactive, reversible FMOS lifecycle edits without per-organ confirmation.
6. **Bounded memory**: Organs promote or prune insight rather than accumulate a shadow project history.
7. **Host adaptation**: GCFMOS learns project conventions without absorbing project-specific policy into its portable kernel.
8. **Safety continuity**: Local skill creation never expands into credentials, publication, accounts, or irreversible external action.

Creational lineage is not an execution dependency. GCFMOS may inspect generated organs, but its portable behavior must not depend on any particular child; a child may not invoke or mutate the parent as a feature authority.

A proposed parent mutation that blurs creator/child roles, centralizes feature knowledge in GCFMOS, hardcodes one host's topology, creates an independent planning system, or lets an FMOS synthesize project priority fails closed. Child organs may provide evidence about creation friction, but they do not directly rewrite the protected kernel.

## Authority Order

Apply this order when instructions or surfaces interact:

1. Safety, approval, and irreversible-action constraints.
2. The user's latest explicit direction.
3. Host-project instructions and canonical truth owners within their domains.
4. Shared capability and deterministic execution contracts.
5. The active FMOS delivery method.
6. Pre-consolidation findings in FMOS `INSIGHTS.md`.

A lower layer may expose divergence in a higher layer but never silently override it. Conflicting canonical owners create an explicit gate, not an implicit decision opportunity for GCFMOS or an FMOS.

## Canonical Truth Boundaries

| Truth role | Host-project owner to discover |
| --- | --- |
| Open outcomes, priority, acceptance state, and gates | Canonical backlog, plan, roadmap, issue set, or equivalent |
| Intended feature contract | Specifications, design records, or equivalent |
| Shipped topology and accepted evidence | Architecture, operations, or release documentation |
| Executable behavior | Code and tests |
| Shared deterministic execution | Host command, script, task, CI, or tool layer |
| Cross-feature method | Capability skills or equivalent reusable protocol layer |
| Feature-local delivery method | FMOS `SKILL.md` |
| Pre-consolidation feature findings | Bounded FMOS `INSIGHTS.md` |
| Completed outcomes | Changelog, release history, closed issue history, or equivalent |

An FMOS links to these owners. It does not mirror their task lists, implementation maps, commands, generated values, decisions, or history.

## Activation Contract

Keeping GCFMOS in a repository is an explicit standing opt-in to a proactive protocol. Whenever an agent receives or discovers a high-level EPIC Feature task, it should evaluate whether the feature needs an FMOS before deep implementation. The agent may create, refine, split, merge, or retire that organ automatically without separate confirmation for each local, reversible skill edit. Composition or generalization into another owner follows that owner's rules.

The user does not need to say “create a skill.” Epic-level intent is the trigger. If evidence remains incomplete but a feature-specific multi-slice feedback loop would cheaply improve continuity, GCFMOS may create a minimal FMOS and let repository evidence evolve it. If maintainers do not want proactive organ creation, they should remove or disable GCFMOS; its discoverable presence constitutes the permission.

This is proactive agent behavior, not a hidden process. GCFMOS runs only while an agent handles repository work; it claims no daemon, timer, watcher, background continuation, or independent execution channel. External publication, credentials, accounts, release actions, and other irreversible effects retain their normal approval gates.

## Project Adaptation

Before generating anything, GCFMOS reads the latest user direction, project protocol, skill graph, canonical work surface, owning specification and architecture, nearby successful skills, script ownership rules, naming style, validation routes, and relevant delivery history.

The templates and heuristics below are defaults, not a universal framework. Project conventions outrank GCFMOS preferences unless they violate a higher safety, authority, portability, or truth boundary.

## Portability Contract

GCFMOS must remain useful when copied with its directory into another project:

- It has no required sibling skill, executable, daemon, registry, provider, CI platform, issue tracker, or repository layout.
- It discovers the host's instruction files, planning surface, documentation owners, skill/plugin location, shared command layer, validation route, discovery mechanism, and release history before acting.
- Names such as `BACKLOG.md`, `/scripts`, `.agents/skills`, and `CHANGELOG.md` are examples, not portable assumptions.
- It generates Feature Meta-Organ Skills through the host's existing extension or skill convention. If none exists, it uses the smallest local convention that makes parent and children discoverable without inventing a second project control plane.
- It references host capabilities through public contracts and never reaches into sibling internals.
- It degrades to an explicit epic outcome when no canonical backlog exists and creates a planning surface only when host conventions and authority permit it.
- It carries no child feature knowledge when moved; only the creational and lifecycle method travels.

Project-specific adapters, templates, examples, and validation commands belong in the host skill graph or generated child, not in GCFMOS's protected kernel. Portability means semantic continuity under a new host, not identical filenames.

## Epic Qualification

A candidate earns an FMOS through a **recurring feature-specific delivery feedback loop**, not through epic size alone. Name that loop in one sentence before materialization.

Treat these as positive signals rather than a rigid checklist:

- The project or user names an EPIC Feature, high-level feature outcome, release-sized capability, or repository reality reveals one coherent multi-slice objective.
- Work will likely cross sessions, agents, subsystems, or independently validated slices.
- The epic has feature-specific routing, evidence, safety, sequencing, reconciliation, or handoff pressure.
- An organ would preserve useful intermediate insight, compress repeated reasoning, and keep procedural detail out of the canonical backlog.
- Existing skills do not already provide a clear home for the same trigger, owners, feedback loop, gates, and retirement condition.

GCFMOS may create a minimal FMOS early and let evidence grow it. It should usually skip a clearly one-shot patch, an idea with no actionable scope, temporary telemetry, folder symmetry, or an epic whose only missing method already belongs to a reusable capability skill. Uncertainty alone does not require user confirmation; cheap retirement keeps early creation reversible.

## Reuse and Idempotent Creation

Before generating an FMOS:

1. Read the project skill graph, canonical epic scope, owning specification and architecture, relevant capability skills, and delivery history.
2. Search existing skills by responsibility, trigger, truth owners, feedback loop, evidence, gates, and retirement condition rather than keyword similarity.
3. Extend an existing organ when the candidate shares materially the same delivery loop and truth owners.
4. Compose an existing capability skill when only one reusable cross-feature method is missing.
5. Split an existing organ when the candidate reveals independently triggered loops with different owners or gates.
6. Create a new FMOS only when a separate organ gives one distinct feedback loop a clearer home.

Treat this tuple as the organ's material identity:

```text
trigger + bounded epic scope + canonical truth owners
+ delivery feedback loop + decisive gates + retirement condition
```

Two organs with materially the same identity are duplicates even when their names or prose differ. Reuse is a compression preference, not a veto on justified creation. Never use numerical semantic-match thresholds as architecture evidence; state the concrete overlap or separation.

Immediately before writing, re-read the canonical epic surface, discovery index, and current repository state. If an equivalent organ appeared, extend or merge it instead of creating a sibling. Register the organ and its discovery change together when the host permits. Concurrent duplicates converge on one organ after preserving any distinct valid method or insight; independent creation history does not justify permanent duplication.

## Creation Protocol

When GCFMOS selects a qualified candidate, no additional creation confirmation is required:

1. Choose a compact role-first name following nearby project conventions, often `<feature>-delivery`; avoid redundant prefixes and architecture claims.
2. Create the child in the host's discovered skill location with valid identity and concise responsibility metadata; for example, `.agents/skills/<name>/SKILL.md`.
3. Define its bounded epic scope, canonical work surface, trigger, truth owners, exclusions, authority boundary, delivery loop, evidence ladder, safety and approval gates, handoff, and retirement condition.
4. Reference shared command and capability contracts instead of copying commands, validators, or sibling internals.
5. Register the organ through the host's canonical skill index or discovery mechanism; for example, `.agents/skills/README.md`.
6. Preserve every open outcome, priority signal, dependency, gate, and acceptance state on the canonical work surface. Move only duplicated procedural guidance into the FMOS and leave stable links between the surfaces.
7. Add bounded `INSIGHTS.md` only when the epic needs temporary feature-local working memory beyond the delivery protocol itself.
8. Run canonical skill metadata, link, context, and completion validation. If the host exposes no validator, apply the semantic fallback below.
9. Report the qualification, reuse decision, material identity, ownership boundary, graph changes, and validation evidence.

### Minimal FMOS Shape

```markdown
---
name: <feature>-delivery
description: <one-sentence feature-delivery responsibility>
---

# <Feature> Delivery

Canonical work surface: <stable link or exact epic outcome>

## Ownership and Authority Boundary
## Trigger and Scope
## Truth Owners
## Delivery Feedback Loop
## Route
## Evidence and Gates
## Working Memory
## Handoff
## Evolution and Retirement
```

Adapt this shape to the project and epic. Add sections only when real constraints require them. Copied feature documentation, a large static checklist, or a mirrored task plan has failed compression.

### Semantic Validation Fallback

If no canonical validator exists, minimally verify:

- identity metadata is valid and the organ is discoverable through the host convention;
- canonical-owner and execution links resolve or have an explicit unavailable gate;
- trigger, scope, owners, exclusions, approval gates, handoff, and retirement are explicit;
- dependency direction preserves the protected kernel;
- the organ creates no task mirror, command fork, stable-truth copy, independent priority system, or hidden execution channel;
- every `INSIGHTS.md` entry has provenance plus a destination or retirement condition;
- the canonical work surface still owns all open outcomes, priority, dependencies, gates, and acceptance state.

Fail closed when identity, ownership, authority, or truth separation cannot be established. Missing optional host tooling alone does not block a direct semantic validation.

## Feature Working Memory

An FMOS may keep `INSIGHTS.md` as bounded pre-consolidation memory for useful feature-local findings that have not yet earned a stable owner. This makes the organ richer than a backlog item without turning it into another backlog.

Good entries capture a finding, its evidence or provenance, why it changes future delivery, and its intended canonical destination or retirement condition. Suitable material includes unresolved design pressure, recurring implementation traps, cross-slice dependencies, falsified assumptions, evidence gaps, and decisions awaiting specification, architecture, backlog, or user promotion.

Do not store raw logs, command transcripts, commit diaries, copied tasks, current generated hashes, or stable facts already owned elsewhere. At each meaningful slice, promote mature truth to its host owner, convert actionable outcomes into the canonical open-work surface, and delete stale or resolved insight. Keep the file small enough to read as one feature checkpoint.

## Slice Selection and Completion Pull

An FMOS does not create feature priority. It may select one bounded slice only from work already eligible under the user's latest direction and the canonical owners' recorded priority, dependencies, gates, exclusions, and acceptance state.

When several slices remain equally eligible and no material product, architecture, risk, scope, release, or resource trade-off exists, the FMOS may prefer the slice that reduces decisive uncertainty or unlocks more canonical work. When selection requires such a trade-off and no owner has resolved it, the FMOS surfaces the exact decision and stops at that gate.

An FMOS should actively pull its epic toward completion without taking priority ownership:

- Keep objective stop conditions and the highest-priority canonically eligible remaining slice visible.
- At relevant checkpoints, report what closed, what remains, and the exact next actionable or gated transition.
- When continuation intent permits, prefer the unfinished epic over unrelated optional polish.
- Recognize completion, consolidate durable knowledge, and stop generating momentum when the objective closes.

Completion pull never overrides the user's latest direction, manufactures urgency, reopens accepted scope, or continues through safety, approval, ownership, or release gates.

## FMOS Operating Contract

A generated FMOS should guide this loop:

```text
verified repository reality
  → refresh and prune the feature working model
    → reconcile against the canonical epic plan
      → select one bounded canonically eligible slice
        → route to owned implementation and shared validation
          → interpret evidence, divergence, and gates
            → promote or prune insight and pull toward completion
```

The FMOS may improve after a slice when a recurring ambiguity, dead end, unsafe shortcut, missing route, or evidence mistake changes future delivery. It must not absorb raw logs, commit diaries, completed task lists, current hashes, or facts already owned elsewhere.

## Evolution Rules

- **Refine** when the same epic exposes a recurring missing decision rule or route.
- **Compose** when a cross-feature capability already owns the required method.
- **Generalize** when concrete reuse pressure shows that a method belongs above one feature; multiple real consumers provide the strongest evidence.
- **Split** when one organ contains independently triggered feedback loops with different owners, gates, or retirement conditions.
- **Merge** when two organs converge on materially the same organ identity.
- **Retire** when the epic closes and no durable feature-specific delivery or operations loop remains.

GCFMOS makes local structural changes from evidence. It does not merge, generalize, or delete organs merely from co-activation counts, age, file size, vocabulary similarity, or speculative future reuse.

## Parent Evolution Protocol

Before changing GCFMOS itself:

1. Ground the mutation in observed friction from creating, evolving, consolidating, or retiring at least one organ, or in a concrete portability failure.
2. Classify the change as portable creational method, host adapter, child-organ method, feature knowledge, or host policy.
3. Keep only portable creational and lifecycle method in the parent; route every other class to its owner.
4. Recheck every protected-kernel invariant, authority layer, and creator/child dependency direction.
5. Read the result as if installed into a project with different filenames, tools, domains, planning conventions, and validation infrastructure.
6. Prefer compression or replacement over accumulating another phase, registry, synchronization layer, or ontology.
7. Validate parent identity, host discovery, links, portability, and context after the mutation.

A large rewrite is acceptable when it preserves the kernel and improves portability or generative usefulness. Small edits are not automatically safe: reject any mutation that gradually turns the parent into a feature coordinator, project manual, priority engine, or universal workflow system.

## Release and Retirement

A release does not automatically preserve or destroy an FMOS. At epic completion:

1. Move implementation truth to the host's code, tests, and owning documentation.
2. Move remaining work and completed impact to the host's canonical open-work and delivery-history surfaces.
3. Move reusable cross-feature method into the owning capability skill only when genuine reuse exists.
4. Promote or retire every remaining `INSIGHTS.md` entry.
5. Keep the FMOS only if the feature retains an ongoing distinct delivery or operations feedback loop.
6. Otherwise remove it and update the project skill graph or discovery surface in the same change.

Do not create parallel organ registries, wisdom stores, sync ledgers, fractal knowledge trees, TTL jobs, or shadow planning systems when the host already has owners for those roles. Reuse the host control plane rather than requiring one preferred file topology.

## GCFMOS Self-Boundary

GCFMOS is a factory and garbage collector for bounded feature-delivery organs, not a super-organ over project delivery. It may evolve its qualification, materialization, validation, and lifecycle method when real organ creation reveals reusable friction. It must not ingest child feature knowledge, coordinate child slices, override their owners, resolve their product decisions, or treat itself as an FMOS.

## Completion Evidence

Report:

- epic and canonical work surface examined;
- recurring delivery feedback loop identified;
- reuse and capability candidates considered;
- qualification and material-identity decision;
- organ created, refined, composed, split, merged, generalized, or retired;
- ownership, authority, and exclusion boundary;
- graph or discovery surfaces updated;
- canonical or fallback validation result;
- unresolved human, owner, safety, release, or external gate.
