# DEOS Framework Delivery History

> Canonical delivery history for the current DEOS repository line.
>
> This repository restarted its own release line at `0.0.0` after the move into the standalone DEOS monorepo. The changelog focuses on shipped outcomes and their impact, not intermediate refactor steps, package-version chores, or implementation diary entries.

## 0.5.4: Proactive Improvement Backlog Anchor

- `[Improvement Loop]` Added, executed, and retired the progressive-improvement lane after moving remaining dependency watches into the externally gated backlog section; removed non-open validation command inventory from `BACKLOG.md` after those entrypoints moved to README/skill documentation; and hardened the backlog audit against command-reference inventory regressions including indented command bullets and validation-entrypoint sections. Impact: autonomous iterations have delivery history here, while `BACKLOG.md` stays focused on future product, runtime, and externally gated work instead of becoming a second changelog or operations reference.
- `[Validation]` Expanded the fast local audit stack in `validate-local.sh --audit-only` and moved the project-specific static audit leaves into the repo-local `alignment` skill: script/Node entrypoints, validation-leaf ownership, template readiness, numeric parsing, simulator determinism, simulator suite/documentation mirror consistency, code suppressions, backlog open-work shape, web-client Domain DAG, and trusted wiki markdown are now checked before heavier CI/build/E2E work; network-backed dependency posture can be added explicitly with `--dependency-audit`. The simulator mirror audit checks both total count and ordered test headings, and the numeric audit rejects direct trimmed-string numeric conversion plus web-client local decimal regex parsing outside the shared parser boundary. Impact: common regressions are caught quickly, the learned gates live with project cognitive infrastructure, and npm registry/audit checks stay opt-in and discoverable.
- `[Scripts]` Hardened local validation behavior by adding missing audit command prerequisites, making `ci-local.sh` run clippy with `-D warnings`, guarding local seeding/native-staking bootstrap numeric environment parsing, defaulting the displayed local web-client port during network bootstrap, dropping the stale pre-DEOS benchmark WASM fallback, and extending template readiness checks to reject pre-DEOS framework artifact-name plus stale TMCTOL-branded wallet-helper regressions. Impact: local tooling fails earlier, matches the zero-warning Rust policy, rejects malformed local-chain setup inputs before mutation paths run, avoids unset optional display variables, and resolves benchmark/browser artifacts through the current DEOS naming line.
- `[Simulator]` Made correctness tests deterministic by replacing ambient randomness with seeded per-test PRNGs, replacing a wall-clock performance assertion with fixed-batch formula execution, adding reward-routing dust conservation coverage, hardening the vesting-cliff math-trap scenario so it cannot silently skip missing floor liquidity, and removing the legacy `@ts-nocheck` suppression. Impact: simulator failures are reproducible while integer conservation, convergence compression, and hot formula paths stay exercised.
- `[Reference Client]` Hardened numeric/domain input handling across token amounts, governance domain ids, reward epochs, proposal item ids, swap slippage, treasury payout assets, internal client ids, and dev signer hex presets; shared numeric/token parsing helpers now enforce complete integer/decimal literals and derive token scale from economics constants. Impact: JavaScript prefix/coercion parsing no longer silently turns malformed UI input into signed actions or misleading logs.
- `[Reference Client]` Stabilized focused UI/architecture seams by wrapping footer status pills on narrow panes, extracting wiki metadata contracts out of `WikiWidget`, splitting governance view-state contracts out of the core governance type module, removing stale TMCTOL-branded signer helper aliases from the browser wallet boundary, exporting the renamed signer/numeric helper contracts through the root barrel, and documenting that `ReadModelValue.fetchedAt` is browser observation/cache time rather than canonical chain time/finality. Impact: small-screen status remains usable, widget/view ownership is clearer, signer helper naming matches DEOS, and read-model provenance stays honest.
- `[Dependencies]` Refreshed the web-client lockfile within range, added a report-only dependency posture audit entrypoint, evaluated the `prettier-plugin-svelte 4.1.0` latest-version path, and rechecked dependency posture. Impact: compatible npm updates are consumed, in-range npm drift, moderate-or-higher audit findings, and unexpected template Cargo update failures now have a repeatable local check, while remaining low-severity findings, formatter peer-range watches, or known template Cargo blockers stay explicitly non-disruptive until a clean upstream path exists and are reported with concrete package names.
- `[Docs/Context]` Promoted table-preservation, simulator determinism, broad suppression discipline, the non-token numeric parser boundary, Domain DAG/wiki trust fast-gate inclusion, honest local bootstrap/seeding boundaries, root validation guidance, and the alignment-skill ownership rule for project-specific audits into durable context/docs while tightening wiki table wording. Impact: future agents and maintainers inherit the rules behind the new gates instead of only seeing one-off script failures.

## 0.4.0: Crystallize AAA And TMCTOL Kernel

- `[AAA]` The actor lifecycle contract was hardened around normal cycles, close tails, manual triggers, timer re-arming, tracked funding, scheduler liveness, canonical close plans, and close-tail finality. Impact: actor behavior is easier to reason about without expanding the AAA feature surface.
- `[AAA Immutability]` System Immutable AAA anchors now protect hard protocol accounts such as the current TOL Bucket A and BLDR Bucket A actors from mutation, pause, close, reopen, or governance/root override. Impact: protocol-anchor actors can be treated as non-discretionary infrastructure.
- `[TMCTOL Guarantees]` The public guarantee contract now distinguishes conditional floor claims, reported-floor inputs, bucket/LP states, burn-liveness classes, Zap postconditions, conservation rules, and conformance status. Impact: economic claims are more precise and less likely to be read as unconditional price promises.
- `[Read Model]` A storage-free `TmctolReadModelApi` exposes bounded live guarantee state from existing runtime state instead of adding consensus analytics storage. Impact: client and light-consumer surfaces can read canonical guarantee state without pulling dashboard/history workloads into the runtime.
- `[Router Fees]` Governance-settable router fees are bounded by an explicit `MaxRouterFee` contract. Impact: Root or governance cannot silently mutate fees beyond the assumptions used by TMCTOL burn-liveness and conservation claims.
- `[Runtime Boundary]` Runtime analytics/dashboard scaffolding was removed from the consensus layer and documented as external indexer, tooling, test-helper, or product-service work. Impact: the runtime stays focused on bounded authoritative state instead of becoming an analytics database.
- `[Staking Rewards]` Phase 1 Fee Sink routing now strengthens the native staking and native staking LP reserve path without donor LP minting. Impact: the current trusted-collator launch line has a concrete pool-level reward route while permissionless-collator reward expansion remains a future boundary.
- `[Operations]` Local operator tooling aligns with the current Polkadot stable2603-2 / node v1.22.2 posture, runtime WASM discovery, and Zombienet startup realities. Impact: local validation and bootstrap flows better match the shipped runtime line.
- `[Backlog]` Completed hardening work was removed from open backlog and captured in delivery history. Impact: `BACKLOG.md` tracks remaining work rather than completed release content.

## 0.3.0: Staking Rework

- `[Native Staking]` The native launch model now centers on liquid `stNTVE`, the canonical `NTVE/stNTVE` AMM, explicitly locked LP nomination, and bounded epoch-based native nomination rewards. Impact: native staking, security liquidity, and reward settlement form one coherent launch contract.
- `[Collator Security]` The old native-binding path was replaced by locked `NTVE/stNTVE` LP nomination with lock, unlock, withdraw, redelegate, valuation, custody, and trusted-collator session behavior. Impact: collator backing depends on explicit liquidity custody rather than liquid receipt balances or cache repair assumptions.
- `[Governance Power]` Native vote power now accounts for locked `$NTVE`, locked `stNTVE`, standalone locked `NTVE/stNTVE` LP, and collator-locked LP through explicit runtime valuation. Impact: already-cast governance weights remain frozen across later exchange-rate, reserve, or custody changes.
- `[Rewards]` Native nomination rewards settle through dedicated claim and compound surfaces while generic same-asset reward claims reject the native staking asset. Impact: native security rewards have a separate bounded lifecycle from generic staking rewards.
- `[AAA Staking]` AAA staking remains portable through generic `Stake`, `Unstake`, and `DonateLiquidity` tasks instead of DEOS-specific native task variants. Impact: downstream runtimes can reuse AAA staking semantics without inheriting one reference chain's native policy.
- `[Liquidity Donation]` Native LP farming can strengthen `NTVE/stNTVE` reserves without minting donor LP, while ordinary compounding still mints and locks LP transactionally. Impact: protocol-owned reserve strengthening and user/operator compounding remain distinct mechanisms.
- `[Read Models]` Bounded canonical views expose native staking exchange rate, canonical reserves, LP issuance, locked-LP/governance custody detail, known reward claimability, and frozen governance power facts. Impact: the reference client can present native staking truth without depending on an indexer for core flows.
- `[Reference Client]` Native staking has a dedicated widget with canonical-chain provenance, exact amount parsing, signer-gated claim/compound actions, LP nomination lifecycle actions, governance custody actions, safe-max helpers, and pending unlock detail. Impact: users can exercise the shipped native staking model from the browser without hidden archive assumptions.
- `[Operations]` Local native-staking bootstrap and guarded activation tooling cover local demo state, plan-only operator calls, readiness probing, and canonical pool/actor sequencing. Impact: operators can prepare native staking flows without ad-hoc transaction assembly.
- `[Governance]` The current governance v1 domain, cadence, and payload-kind baseline is closed while wider parameter, observability, browser-composition, and archive growth remain future-gated. Impact: the launch governance surface is bounded instead of open-ended.
- `[Archive Boundary]` Governance archive search and ballot timelines are explicitly materialized-provider work rather than implied by bounded on-chain retained outcomes. Impact: the client read-model boundary remains honest.
- `[Docs]` Staking specs, architecture docs, wiki pages, and backlog state align with the final native staking and governance-product baseline. Impact: stale native-binding and `StakeNative` wording no longer contradicts the shipped model.

## 0.2.0: Runtime Hardening

- `[Staking Maintenance]` Staking `on_idle` maintenance now respects remaining-weight budgets across native-delegation ingress, dirty-cache repair, reward ingress, and low-budget truncation cases. Impact: background staking maintenance stays bounded under constrained block budgets.
- `[Staking Ranking]` Collator ranking uses bounded cached delegated-native shares with exact-read fallback, two-phase dirty-cache repair, stale zero-exposure retirement, and immediate clean-cache updates. Impact: candidate ordering remains deterministic and candidate-count-bound even with many bindings.
- `[Reward Ingress]` Reward ingress is event/domain indexed, budget-aware, resumable, and guarded by bounded scans and reverse indexes for receipts and governance reward domains. Impact: reward processing avoids broad unrelated-pool scans while preserving settlement correctness.
- `[TMC]` Curve creation and mint distribution now preflight asset existence and use transactional mint/distribution behavior. Impact: late mint failures cannot strand collateral transfers or partially mutate mint state.
- `[TMC Tests]` Runtime and pallet coverage now asserts mint success/failure behavior, sink topologies, rollback paths, native burn accounting, conservation, and benchmarked mint-with-distribution behavior. Impact: TMC economics are defended by concrete runtime and pallet regressions.
- `[XCM]` Runtime XCM origin conversion and barrier behavior are covered for parent, sibling, `AccountId32`, passthrough, paid/unpaid paths, and queue-controller access. Impact: sovereign `Transact` ships fail-closed through an intentionally empty safe-call matrix.
- `[Governance Treasury]` Tactical `L2TreasurySpend` is documented as one bounded source topology rather than a generic treasury-spend family. Impact: governance execution authority stays honest about current funding surfaces.
- `[Router]` Axial Router swaps now pre-validate gross input affordability and route fees transactionally before execution across XYK, mint, and multi-hop paths. Impact: fee failures cannot execute routes or mutate user/pool balances.
- `[Router Tests]` Router integration and benchmark coverage now assert balance deltas, fee-sink deltas, event ordering, native-input semantics, repeated fee accumulation, failed-swap no-fee behavior, and non-zero benchmark output. Impact: routing behavior is tested as economics, not just API compatibility.
- `[Weights]` Router, staking, TMC, and AAA task-weight surfaces were regenerated or adjusted after the hardened paths landed. Impact: runtime weights better reflect the shipped storage and execution patterns.
- `[Docs]` Core and router architecture docs now describe fail-closed XCM, bounded tactical treasury execution, gross-debit swap preflight, and fee-before-execution ordering. Impact: architecture mirrors the hardened runtime behavior.

## 0.0.0: Delivered Baseline

- `[Runtime Platform]` The repository ships a Polkadot SDK 2603 / Omni Node-ready runtime baseline with CI, local tooling, benchmark lanes, and deployment assumptions aligned to the current framework line. Impact: DEOS starts from a modern parachain-template posture instead of a custom-node boilerplate path.
- `[Framework Identity]` DEOS is established as the forkable framework identity, while TMCTOL is the first economic standard running on top of it. Impact: downstream forks can reuse the substrate without inheriting one product narrative as the framework name.
- `[Economic Kernel]` AAA, Axial Router, TMC, TOL buckets, Asset Registry, and token-driven runtime patterns form the initial deterministic economic kernel. Impact: issuance, liquidity, routing, automation, and asset identity are implemented as protocol services rather than ad-hoc off-chain coordination.
- `[Governance]` Governance v1 ships bounded dual-track domains, payload-kind/cadence metadata, active/finalized query surfaces, advisory submission, runtime-upgrade authorization, and tactical invoice-native treasury governance. Impact: the baseline has explicit governance authority without importing an unbounded OpenGov surface.
- `[Staking]` Share-vault staking ships with multi-asset `stXXX` receipts, native `stNTVE` support, sparse reward ingress, same-asset auto-compound settlement, and governance-conditioned reward export. Impact: staking is a reusable economic substrate rather than a single native-token special case.
- `[Reference Client]` The browser client covers governance, wallet, swap, pane layout, execution feedback, bounded runtime truth, and generated wiki consumption. Impact: the framework has a local product/reference surface for testing user-facing flows.
- `[Docs]` The documentation plane is typed around specifications, architecture mirrors, generated wiki pages, glossary/navigation metadata, context files, backlog, and delivery history. Impact: protocol contracts, implementation maps, onboarding, open work, and completed work have separate homes.
- `[Validation]` Simulator tests, Rust checks, benchmark compilation, web-client checks, local probes, and operator scripts form the initial evidence-driven validation stack. Impact: runtime and frontend changes can be judged against explicit gates instead of informal confidence.
